# B-001: /crm pipeline shows BLANK hub names + booking slugs for every hub
Severity: S2 major
Found by: TR-009 while testing T-009 (Journey B, CRM → provision → handoff)
Status: verified-closed

## Repro steps
1. Sign in to https://amityx.pages.dev/login as any active `crm_admin` (e.g. the
   seeded founder, or a throwaway admin with a `crm_admins` row).
2. Open `/crm/hubs`.
3. Observe the pipeline table's **Hub** column.

## Expected vs actual
Expected: each row shows the hub's business name (a link) + city/state; searching by
business name filters the list; the hub-detail page shows the hub name.
Actual: the **Hub** name column is EMPTY for every row (10 seeded pilots + any
CRM-/self-provisioned hub). Search by business name returns "No hubs match your
filters" (0 shown) even though 11 hubs are active. Admins can only tell hubs apart by
owner email. `/crm/hubs/:hubId` likewise renders a blank name.

Root cause: `ApiRepository.listCrmHubs`/`getCrmHub` read the hub name via a PostgREST
embed `crm_hub_profiles.select('*, hubs(name, slug, city, state, plan, created_at)')`.
The `hubs` root row is gated by RLS policy `hubs_read` → `auth_read_hub_ids()` = a
user's member hubs UNION active support-granted hubs. A platform admin is neither a
member nor (by default) support-granted for any hub, so the embed resolves to `null`
and `toCrmHubListItem` maps `hub.name ?? '' → ''`. Every other CRM field
(owner_email, stage, status, follow-ups, notes) comes from `crm_hub_profiles`, which
admins CAN read, so the breakage is limited to the hub root fields (name/slug/city/
state/plan/created_at) — but those include the pipeline's primary identifier.

## Evidence
- Live snapshot (`/crm/hubs`, signed in as a fresh crm_admin): header "11 active",
  first data row renders `— Free Prospect — Open` (leading `—` is the owner column;
  the Hub column link is empty).
- Node probe with a throwaway crm_admin JWT running the exact `listCrmHubs` query:
  every row returns `embeddedHub: null`; a direct `hubs` select returns 0 rows.
- Policy confirmation: `hubs_read USING (id IN (SELECT auth_read_hub_ids()))`;
  `auth_read_hub_ids()` = member ∪ active support grant (neither holds for an admin).

## Fix design
Add a least-privilege RLS SELECT policy so an active platform admin can read the hub
ROOT row (name/slug/city/state/plan/timezone/flags) — pipeline metadata the CRM needs
— WITHOUT widening access to any tenant DATA table (children/notes/photos/attendance/
guardians stay member-/support-gated, unchanged). New migration
`supabase/migrations/20260713150000_crm_hub_read.sql`:
`CREATE POLICY hubs_crm_read ON hubs FOR SELECT TO authenticated USING (public.is_crm_admin());`
No app/bundle change (so no redeploy); no other policy touched. Regression net grown:
two new adversarial checks assert the admin CAN read hubs metadata yet STILL cannot
read tenant data without a grant.

## Fix verification (tester fills)
2026-07-13 — Migration applied to prod (Supabase Management API). Node probe re-run:
crm_admin now reads all 11 hub names via the embed; direct hubs read returns rows.
`/crm/hubs` list + search now show real hub names (Journey B green). RLS adversarial
suite re-run GREEN at 100/100 (was 98; +2 new hubs-metadata checks) — isolation of
tenant data unchanged (admin still blocked from children/notes without a support grant).
