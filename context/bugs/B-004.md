# B-004: "Forgot your password?" looks broken after a failed sign-in attempt
Severity: S2 (confuses a core auth flow — founder-reported from production)
Found by: founder, live on https://amityx.pages.dev
Status: verified-closed

## Repro steps
1. Open https://amityx.pages.dev/login.
2. Enter a real account's email + a WRONG password. Click "Sign in" — see the
   "That email and password don't match..." error banner.
3. Click "Forgot your password?".

## Expected vs actual
Expected: the reset email sends and the UI clearly confirms it (P.9 rule 8 — one
clear result, forgiving errors).
Actual: the reset email genuinely sent (reproduced via Playwright against production:
the `/auth/v1/recover` request fired correctly with the right redirect and returned no
error) — but the OLD "email and password don't match" error banner from step 2 was
NEVER cleared, so it rendered on screen at the same time as the new "Check your email
for a link..." success notice. Two contradictory messages at once reads as "this didn't
work" even though it did.

## Evidence
- Playwright reproduction against production confirmed both `[role="status"]` (new
  success notice) and `[role="alert"]` (stale wrong-password error) visible
  simultaneously after the click.
- `app/web/src/pages/marketing/Login.tsx`: `forgotPassword()` and `resendVerification()`
  called `setNotice(...)` but never `setError(null)`/`setErrorKind(null)`.

## Fix design
Clear `error`/`errorKind` at the start of both `forgotPassword()` and
`resendVerification()`, before setting the new outcome — so a prior sign-in error can
never linger next to (or in place of) the current action's own result. Added a separate
`actionBusy` state (distinct from the sign-in button's `busy`) so these two secondary
actions show correct "Sending…" feedback and can't double-fire, without mislabeling the
primary "Sign in" button as busy during a forgot-password/resend request.
