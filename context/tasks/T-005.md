## Spec
The 18 tables per ARCHITECTURE.md §Data model: hubs, hub_members, programs, class_sessions, children
(photo_consent), guardians, child_guardians, enrollments, booking_requests, announcements,
photo_moments, attendance, child_notes (visibility fields reserved), guardian_links, crm_admins,
crm_hub_profiles, crm_followups, crm_comm_log. Every tenant table carries `hub_id` NOT NULL.
RLS ENABLE + FORCE on every table; deny-by-default (no policy ⇒ no access). Policy patterns:
- Tenant tables: membership via a `SECURITY DEFINER` helper (e.g. `auth_hub_ids()` reading
  hub_members for `auth.uid()`); CRUD gated on `hub_id IN auth_hub_ids()`; hub_staff must NOT
  touch billing/settings columns (separate policy or column privileges).
- crm_*: gated on membership in crm_admins (platform staff only); NOT reachable by any hub role.
- booking_requests: anonymous INSERT-only policy (public page); NO anon SELECT; a per-hub/day
  insert rate limit (DB trigger) blunts spam; owner reads via hub RLS.
- guardian_links: the ONLY parent read path — token resolved by a `SECURITY DEFINER` RPC that
  verifies signature + not-expired + not-revoked, scopes to that guardian's children, and drops
  any child without photo_consent. No broad anon RLS on child tables.
- Platform support-access into a hub = explicit, time-boxed grant row + append-only audit log.
