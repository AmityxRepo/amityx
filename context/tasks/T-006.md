## Spec
Supabase Auth (email+password, plus email OTP as a passwordless option); custom SMTP = Google
Workspace help@agapaycare.com configured in the Supabase dashboard (env NAMES only, never in app
code). Signup wizard, each step one job (P.9): (1) owner account + email verify → (2) hub creation
(name → auto-slug, collision-checked) → (3) program-template picker (art, swim, karate, day care,
boot camp, open-play, camp; multi-select with sensible pre-checks) → (4) first class schedule →
(5) optional staff invites by email. Wizard progress persists so a drop-off resumes where it left
off (protects the ≤15-min criterion). On success, land on a populated hub dashboard. CRM handoff:
signup atomically creates the hub's crm_hub_profiles row (onboarding_stage seeded). Session persists
+ auto-refreshes (supabase-js). Error behavior (P.9 rule 8 — never a dead end): duplicate email →
"That email already has an account. Sign in?"; weak password → inline rule; slug collision →
auto-suffix + editable; SMTP send failure → retry + surface the verification path. Staff invitees
are provisioned WITHOUT billing/settings access (enforced by T-005 RLS).

The founder saved custom SMTP in the Supabase Dashboard (smtp.gmail.com:465, sender
help@agapaycare.com). Ran a live test: (1) fresh signup, (2) resend against a genuinely unconfirmed
user (created via admin API — resend against a phantom email returns a false-positive 200 in ~1ms
without attempting SMTP, so this must be tested against a real unconfirmed user), (3) one attempt
to the founder's own inbox via a `noel.adv.castillo+amityxsmtptest@gmail.com` alias (delivers to
his real inbox; his actual `noel.adv.castillo@gmail.com` account is already a confirmed CRM admin
seeded by T-008 and was NOT touched).

**Result: 0/3 pass.** All three fail identically and immediately with `500
{"error_code":"unexpected_failure","msg":"Error sending confirmation email"}`. Pulled the
underlying error from Supabase's auth logs via the Management API — it is unambiguous:
`535 "5.7.8 Username and Password not accepted... BadCredentials" - gsmtp`. This is Gmail rejecting
the SMTP credentials Supabase has on file — consistent across every attempt (not a transient
platform issue; a real hiccup would not reproduce the identical error 3/3 times). No email reached
any inbox — the founder's inbox check does not apply yet. Supabase does not persist a user row when
the confirmation-send fails, so no cleanup was needed; verified the auth.users list is unchanged
(same 3 pre-existing accounts, nothing added).

**Exact fix needed:** the password field in Supabase Dashboard → Auth → SMTP Settings is wrong for
that username. Gmail/Workspace SMTP relay requires a 16-character **App Password** (not the regular
account password) — generated at myaccount.google.com/apppasswords, which itself requires 2-Step
Verification to be enabled on the help@agapaycare.com Workspace account first. Likely causes: (a) the
regular account password was entered instead of an App Password, (b) the App Password was
mistyped/truncated, or (c) 2-Step Verification isn't enabled so no valid App Password could exist
yet. Re-run `npm run test:smtp` after correcting it — it is a durable, reusable check, not a
one-off script.
