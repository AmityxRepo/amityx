Design decisions
- Group-photo consent = REJECT AT WRITE (option a): create_photo_moment refuses any photo whose tagged set
  includes a no-consent/unknown child and names the blocker; makes the read invariant airtight (a distributed
  photo has zero non-consented subjects — no runtime filter to get wrong).
- Signed-URL pattern = EDGE FUNCTION, not anon-SELECT capability URLs: a Storage SELECT policy broad enough to
  SIGN a known object is also broad enough to LIST/enumerate the whole bucket (same RLS gate) — unacceptable for
  child media. Bucket stays fully private; the function re-derives allowed paths from the token and signs with
  the service key. Still $0.
- Purge = GitHub Actions cron + Node script, not pg_cron: pure SQL can only delete storage METADATA, orphaning
  the S3 bytes; the script uses the Storage API to remove bytes AND rows together (defense-in-depth: the feed
  already filters expired photos before purge runs).
- Hub-wide "announcement image" = `announcements.image_path` (general/flyer photos, shown to all families,
  no child tags); any photo OF a child goes through the consent-enforced photo path.
