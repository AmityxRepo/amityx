## Spec
Guardian access via signed scoped expiring links (guardian_links) sent by email/SMS-able URL —
NO account, NO install. Parent view (PWA-installable anyway): child's schedule, announcements,
photo moments. Staff capture: snap/upload → compress client-side (~200KB webp) → storage ADAPTER
(stage 1 Supabase Storage; stage 2 R2 per D-011 — interface designed now, flip = config).
Per-child photo_consent enforced at write AND read (no-consent child never appears in group-photo
distribution). 30-day rolling window purge job (free tier); all media URLs signed + expiring.
Announcements: aggregate view counts only.

## Spec — clarifying
guardian_links = signed, scoped, expiring, REVOCABLE token in the URL (no account, no install);
resolved only by the T-005 `SECURITY DEFINER` RPC, which scopes to that guardian's children and
drops any child without photo_consent. Parent view (PWA-installable but not required): that child's
"Classes", "Updates" (announcements), "Photos". Staff capture in /app: snap/upload → compress
client-side to ~200KB webp → write via a storage ADAPTER interface (`put` / `getSignedUrl` /
`delete`) selected by env `MEDIA_BACKEND` — stage 1 = Supabase Storage (private bucket), stage 2 =
R2 (D-011); backend swap is config, not code. photo_consent enforced at WRITE (tag subjects) AND
READ (a no-consent child never appears in any parent-facing surface, incl. group photos). ALL media
served via signed, expiring URLs — never a public bucket. 30-day rolling purge: a scheduled job
(pg_cron OR GitHub Actions cron → RPC/edge function; free) deletes bytes + metadata past the window
(D-009). Announcements store AGGREGATE view counts only — NO per-recipient receipt rows (R-003).
Link delivery = email (Workspace SMTP) or a human-pasted SMS-able URL (D-014); no system SMS.
NOTE: this ships NO AI/LLM (D-004) — no lethal-trifecta break applies; the security surface here is
token scoping + consent enforcement + signed URLs.
