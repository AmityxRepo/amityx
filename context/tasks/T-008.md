### Schema addition (small, additive — see migration file header for full rationale)
`supabase/migrations/20260712221500_crm_pipeline_extensions.sql`, applied live:
1. `crm_onboarding_stage` gets `'prospect'` (before `'signup'`) — required by the spec's seed
   (`onboarding_stage: 'prospect'`) but absent from T-005's enum.
2. `crm_hub_profiles.archived`/`archived_at` — required by the spec's reversible archive toggle;
   absent from T-005's table.
3. Two new SECURITY DEFINER RPCs, `crm_provision_hub` + `crm_invite_hub_owner` — the literal reuse
   of T-006's `provision_hub`/`create_hub_invite` isn't reachable without violating ARCHITECTURE's
   binding rule ("platform_admin … never inside hub data by default"): `provision_hub` binds
   `hub_members` ownership to `auth.uid()` (the calling CRM admin, permanently and un-audited) and
   `create_hub_invite` only ever mints STAFF invites. `crm_provision_hub` creates the
   hub+programs+crm_hub_profiles row identically to `provision_hub` but grants the admin NO
   membership; `crm_invite_hub_owner` mints an OWNER-role invite, guarded to hubs with zero members
   (never displaces a real owner). `accept_hub_invite` (T-006, untouched) already assigns whatever
   role the invite carries, so the end state (hub + programs + crm_hub_profiles + hub_members owner)
   is DATA-IDENTICAL to a self-signup hub, per the spec's "identical to a self-signup hub" wording.
   Flagging this as a judgment call grounded in an already-decided architecture rule, not a guess.
