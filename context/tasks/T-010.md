## Spec
Public route per hub (amityx.pages.dev/h/{slug}): hub profile, program cards with schedules,
capacity/waitlist state, booking-request form (child name+age, guardian contact — minimal PII,
no account required; rate-limited, spam-protected). Request → owner notification (email) +
/app booking inbox. "Powered by Amityx" footer (the P.8 loop). SEO-sane (static-ish render),
shareable link for the hub's Instagram bio.

## Spec — clarifying
Public, unauthenticated route `/h/{slug}`: hub profile, "Activities" cards with "Classes" +
schedules, capacity/waitlist state, and a booking-request form (child first name + age band,
guardian name + contact — minimal PII, NO account). Anti-spam WITHOUT any paid captcha or paid SMS:
hidden honeypot field + minimum fill-time + the per-hub/day insert rate limit from T-005; a tripped
honeypot is rejected silently. On submit → insert booking_request → owner notified by EMAIL via
Workspace SMTP (D-014: NO system SMS in V1; the shareable link is "SMS-able" by a human) → appears
in the /app "Requests" inbox. Anonymous visitors can INSERT only; never SELECT roster or other
requests (T-005 RLS). "Powered by Amityx" footer (the P.8 loop). SEO: per-hub title/OG meta,
static-ish first paint. Edge cases: unknown slug → friendly "page not found" (no stack/leak); full
class → "Join waitlist"; duplicate submit → idempotent-friendly thank-you, no error wall.

## Result
**Result: done.**

**Summary:** Closed the anon-read gap with a `get_public_hub_page(slug)` SECURITY DEFINER RPC
(mirrors `resolve_guardian_link`'s curated-field pattern) exposing only hub name/city/state/
address/timezone + active programs + upcoming sessions + live capacity counts — never
plan/stripe_customer_id/settings. Built `/h/{slug}`: single-scroll, no-nav public page (activity
cards with schedule + "Join waitlist" when full, booking form: child name + age band + guardian
contact), honeypot + min-fill-time anti-spam (silent drop, never tips a bot), per-hub `<title>`/OG
meta via a small CSR helper (JS-only — crawlers that don't execute JS still see the site-wide
default, same limitation every route in this SPA already has), "Powered by Amityx" footer.
Age band stored in `booking_requests.message` (not fabricated into `child_birthdate` — an
approximated birthdate would silently corrupt the child record if the request is later accepted;
owner can ask for the real birthdate). Unknown slug and a `public_booking_enabled=false` hub both
return the identical `not_found` shape (no enumeration).
