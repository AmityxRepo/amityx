# R-003: Free-tier capacity model — how far before paid Supabase? (2026-07-11)
Task: founder question · Method: analysis against verified limits (R-002) + V1/P.8 usage model.
All figures are estimates; re-baseline with real telemetry after the first 5 hubs.

## Reference hub (assumption set)
100 enrolled children · 150 guardians (~60 monthly-active viewers) · 5 staff · 20 class
sessions/wk · 20 photos/wk (compressed ~200KB @1280px webp) · 2 announcements/wk · kiosk check-ins.

## Consumption per hub (with D-008 architecture: media on R2, JSON on Supabase)
| Resource | Per hub | Free limit | Hubs supported |
|---|---|---|---|
| Supabase DB (rows+indexes: ~10k attendance/yr @~300B, ~2.5k notes, ~1k sessions, misc) | ~6–10 MB/yr | ~450 MB usable of 500 MB | **~45–75 hub-years → 35–60 hubs w/ 1yr history** |
| Supabase egress (JSON only — rosters, schedules, announcement text) | ~100–250 MB/mo (60 active parents × 0.5–2 MB + staff 5 × 10–20 MB) | 5 GB/mo | **~25–50 hubs** ← binding constraint |
| Supabase auth MAU (staff + owner; parents mostly link-based in V1) | ~10–70 | 50,000 | not binding (>500 hubs) |
| R2 media storage (30-day rolling window ≈ 85 photos live) | ~18 MB steady | 10 GB | ~500 hubs (even keep-everything ≈ 50 MB/hub/yr → 200 hub-years) |
| R2 egress (parent photo views: 60 parents × 10 views/wk × 200KB ≈ 0.5 GB/mo/hub) | ~0.5 GB/mo | **$0 — R2 egress is free** | unlimited (ops quotas: 1M writes/10M reads/mo ≈ >100 hubs) |

## Counterfactual (why D-008 matters): photos in Supabase Storage instead
Photo delivery (~0.5 GB/mo/hub) hits the 5 GB/mo egress cap at **~8–12 hubs**, and the 1 GB
storage cap at ~20–50 hubs even with the rolling window. R2 removes both at $0.

## Bottom line
- **Capacity ceiling on $0 infra: ~25–50 active hubs ≈ 2,500–7,500 registered parents
  (~1,500–3,000 monthly-active), bounded by Supabase JSON egress, then DB size.**
- The P.6 gate (paid conviction by ~10 hubs) fires LONG before capacity does → in practice the
  Supabase Pro upgrade ($25/mo) will be triggered by duty-of-care (daily backups, no pausing)
  at the first paying hub — never by running out of free capacity.
- 7-day pause risk in quiet early weeks: free GitHub Actions cron pinging a health endpoint
  keeps the project awake (T-009).
- Watch in telemetry: photo posting volume (if hubs post 50+/wk the window math shifts) and
  announcement fan-out design (store aggregate reads, not per-recipient receipt rows).
